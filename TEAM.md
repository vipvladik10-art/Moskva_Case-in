# Состав команды и распределение ролей

Команда: **5 человек** — 2 программиста + 3 инженера.
Документ фиксирует роли, зоны ответственности и матрицу RACI для всех задач из
[ROADMAP.md](./ROADMAP.md).

> **Правило одного владельца**: у каждой задачи ровно один Responsible (R).
> Все остальные — Accountable (A), Consulted (C) или Informed (I).

---

## 1. Роли

### P1 — Backend-программист (тимлид по коду)
- FastAPI, асинхронные адаптеры к метео-API, бизнес-логика, БД.
- Владеет CI и Docker-образами бэкенда.
- Финальный ревью всех PR в `backend/`.

### P2 — Frontend-программист
- React + Vite + TypeScript, MapLibre/Leaflet, мобильная адаптация.
- Дашборд диспетчера и интерфейс мастера участка.
- Push-уведомления (Web Push API / Firebase для демо).

### E1 — Инженер-технолог (дорожное строительство)
- **Главный носитель домена.** Переводит ГОСТ и техкарту в требования.
- Формулы, температурные режимы, нормативные ограничения.
- Acceptance-критерии для алгоритмов «зелёное окно» и «Успеть до дождя».
- Валидация выходных данных (что система рекомендует — соответствует ли ГОСТ).

### E2 — Инженер данных / ML
- Подбор и сравнение метео-API, лимиты, ключи, fallback-цепочки.
- Сервис нормализации погодных данных, ансамблирование прогнозов.
- (Опц.) ML-модель коррекции прогноза по историческим расхождениям.
- Готовит mock-датасеты для демо и нагрузочных тестов.

### E3 — Системный инженер / DevOps & QA
- Docker Compose, инициализация PostGIS/TimescaleDB, миграции.
- CI (GitHub Actions): линт, тесты, сборка образов.
- Интеграционное и нагрузочное тестирование, симуляция «дождя».
- Презентационный стенд для финала, демо-данные, видеозапись прогона.

## 3. Каналы коммуникации

- **Дейлики**: 15 минут, ежедневно в 10:00 (онлайн).
- **Демо спринта**: каждые 2 недели, пятница, 17:00.
- **Канал**: Telegram-чат + #standup; критичные баги — голос.
- **Доска задач**: GitHub Projects (см. `.github/` шаблоны Issue).
- **Документация**: всё в `docs/`, Markdown, через PR.

---

## 4. Definition of Done

Задача считается выполненной, когда:

1. Код в `main` через PR, прошёл ревью владельца модуля.
2. Зелёный CI (lint + тесты + сборка образов).
3. Покрытие unit-тестами критичных формул ≥ 80 %.
4. Обновлена документация в `docs/` (если задача затрагивает контракт/формулу).
5. E1 подтвердил соответствие требованиям ГОСТ (для задач с расчётами).
6. E3 проверил, что docker compose поднимается «с нуля».
